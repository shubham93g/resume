# Resume

[![CI](https://github.com/shubham93g/resume/actions/workflows/ci.yml/badge.svg)](https://github.com/shubham93g/resume/actions/workflows/ci.yml)

Live at: https://shubham93g.github.io/resume/

An HTML resume designed to work well in three contexts: browser (desktop + mobile), print (A4), and PDF download. Each imposes different constraints, and a few of the decisions made to balance them are worth documenting.

## Technical notes

### Print layout

The layout targets exactly 2 A4 pages — this is a hard constraint, not a guideline. Dark mode is suppressed at print time and colours are forced to a white background. Adding vertical space anywhere (padding, margins, new lines) requires reclaiming it elsewhere; the layout doesn't have slack.

### In-page controls

The page has a theme toggle (dark/light) and font-size adjustment controls. Dark mode defaults to the OS setting (`prefers-color-scheme`); the toggle persists the choice to `localStorage`. The initialisation order is: URL param → `localStorage` → system preference — so a fresh visit with no params respects whatever the OS is set to, a returning user gets their last choice, and a URL param overrides both without writing anything back to storage.

`?theme=dark|light` and `?font=N` (integer, `-2` to `+4`) are the supported URL params. Invalid values fall back silently rather than erroring. The PDF generation step uses `?theme=dark&font=1` to get a consistent render regardless of what CI's default system theme would be.

### PDF generation

The PDF is generated by Puppeteer and served via GitHub Pages — it is not committed to the repository. Two settings matter for getting a usable PDF:

- `tagged: true` tells Chromium's PDF engine to produce a tagged PDF with a proper text layer, which is what preserves the ToUnicode CMap tables for self-hosted TTF fonts.
- `waitUntil: 'networkidle0'` ensures the self-hosted fonts have finished loading before the page is captured.

### CI/CD

A single workflow (`ci.yml`) runs three jobs sequentially: `validate → generate-pdf → deploy`. The last two are skipped on PRs; only `validate` runs, and it's the required branch protection check. Shared Puppeteer setup (Node install, Puppeteer install, local HTTP server) lives in a composite action at `.github/actions/puppeteer-server/` and is reused by both `validate` and `generate-pdf`. Puppeteer launch options and PDF settings are consolidated in `.github/scripts/puppeteer-config.js` so there's one place to change them.

### Validation

The `validate` job runs a Puppeteer-based test suite against a local server before the PDF is generated or the site is deployed. It checks:

- **URL params** — `?theme=dark|light` applies the correct theme; invalid values (`?theme=invalid`) don't crash and resolve to a valid theme. `?font=N` applies the correct offset; out-of-range and non-numeric values fall back within bounds.
- **Page count** — the PDF is exactly 2 pages. This is tested by generating the PDF and counting `/Type /Page` objects in the raw output, which catches regressions from layout changes before they ship.

Results are written to the GitHub Actions step summary as a table, so failures are visible without digging through logs.

### Font embedding

IBM Plex Sans is self-hosted as TTF files (`fonts/`) rather than loaded from Google Fonts. The reason is specific to PDF generation: Chrome's PDF engine embeds WOFF2 fonts with incomplete ToUnicode CMap tables, which breaks text selection and copying in some PDF viewers — macOS Preview being the most common one. TTF embedding preserves the full character mapping, so text in the generated PDF stays selectable. Self-hosting also means the font format is never converted on the way to the PDF engine.
